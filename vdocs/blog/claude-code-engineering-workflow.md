# Claude Code 工程化工作流：从需求到 PR 的可复制路径

很多团队把 Claude Code 当“高级补全”，结果不稳定。  
更有效的方式是把它嵌进交付流程，而不是单点提问。

## 1. 四阶段工作流

1. 需求结构化
2. 变更执行
3. 验证闭环
4. PR 交付

## 2. 阶段一：需求结构化（决定上限）

输入建议固定为：

- 目标：最终可验收结果
- 边界：允许改哪些文件，不允许动哪些模块
- 约束：版本、兼容性、风格、性能红线
- 验收：必须通过的命令与行为

如果输入模糊，后续所有“修补”都是高成本。

## 3. 阶段二：变更执行（先通路再优化）

推荐顺序：

1. 先补数据/路由通路
2. 再补页面与交互
3. 最后做样式与重构

每一步都要求输出：

- 改了哪些文件
- 为什么这样改
- 可能影响哪些路径

## 4. 阶段三：验证闭环（证据先于结论）

至少跑三类验证：

- 静态检查：lint / type-check
- 单元验证：test
- 交付验证：build / preview

让 Claude Code 输出“命令 + 关键结果摘要”，而不是只说“已通过”。

## 5. 阶段四：PR 交付（可审阅、可回滚）

PR 描述建议固定模板：

```text
Purpose:
Key Changes:
Verification:
Risk:
Rollback:
```

## 6. 结论

Claude Code 的价值不在“写得快”，而在“快且可验证”。  
把它放进工程流程，才是稳定产能。
