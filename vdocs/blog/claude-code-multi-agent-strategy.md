# Claude Code 多代理协作：任务切片、并行执行与结果合并

多代理不是“多开几个窗口”，关键是切片策略。

## 1. 什么时候该并行

适合并行的任务：

- 互不依赖的读操作：代码扫描、配置梳理、接口盘点
- 写入范围互斥的实现任务：A 改路由，B 改组件，C 改测试
- 与主线并行的验证任务：性能对比、回归检查

不适合并行：

- 同一文件的高耦合改动
- 结果必须串行决策的关键路径

## 2. 切片原则

1. 明确所有权：每个代理负责哪些文件
2. 明确输出物：不是“分析一下”，而是“提交可合并变更”
3. 明确完成条件：必须可验证

## 3. 推荐编排模型

1. 主代理：负责任务拆分与最终集成
2. 执行代理：按模块改代码
3. 校验代理：独立运行测试/构建/风险扫描

主代理不重复做子代理已完成的工作，只做整合与冲突消解。

## 4. 合并策略

- 先合并低风险、低耦合 patch
- 再合并高风险改动
- 每合并一步都跑最小验证

## 5. 常见失败模式

- 切片按“功能名”而不是“文件边界”
- 多代理都改同一核心文件
- 忽略公共类型/工具变更的连锁影响

## 6. 结论

多代理提升的是吞吐，不是正确性。  
正确性仍来自：清晰边界、可验证输出、稳定合并策略。
