# 前端架构决策手册：不是“最先进”，而是“最匹配”

很多架构问题不在技术，而在决策方式。

## 1. 架构先回答 3 个问题

1. 这个页面的核心目标是 SEO、转化，还是效率？
2. 主要瓶颈是首屏、交互，还是协作成本？
3. 团队能否长期维护这套复杂度？

## 2. 常见场景与推荐组合

## 2.1 内容与营销站

- 渲染：SSG/SSR
- 交互：Islands/局部 hydration
- 数据：CDN + 轻量 BFF

适合：需要搜索曝光和高首屏稳定性的页面。

## 2.2 后台与中台

- 渲染：SPA
- 数据：BFF/网关聚合
- 状态：服务端状态与本地状态分层

适合：重交互、低 SEO 诉求、复杂权限。

## 2.3 多团队并行系统

- 组织：模块化优先，微前端谨慎
- 共享：Design Token / 组件契约 / API 契约
- 发布：主应用编排 + 子模块灰度

适合：组织复杂度高于页面复杂度的团队。

## 4. 架构升级节奏

1. 先局部试点，不要全站迁移
2. 先做度量，再做优化
3. 先建回滚方案，再上线新架构

## 5. 结论

好架构不是“技术栈清单”，而是“决策质量 + 可回滚能力”。
